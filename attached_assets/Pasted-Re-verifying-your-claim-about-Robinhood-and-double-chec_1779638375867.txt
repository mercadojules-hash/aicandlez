Expanded Registry RCA (covers your new capability matrix)
Gap 1 — Status enum is too narrow
catalog.ts line 39:

status: "live" | "beta" | "coming_soon";

You're asking for: enabled | disabled | beta | coming_soon | admin_only | hidden | region_restricted. These conflate two orthogonal concepts. Cleanest model:

// What the implementation is
maturity: "live" | "beta" | "experimental" | "coming_soon";
// Independent boolean flags (each defaults sensibly)
enabled:           boolean;   // master switch  — false = hide everywhere
adminOnly:         boolean;   // only admin/super-admin sees + connects
customerVisible:   boolean;   // shown in customer modal (false = hidden but admin can still see)
regionRestricted:  string[] | null;  // ["US"] = blocked for US users, null = global
requiresLiveGate:  boolean;   // honors Task #165 customer paper-only gate

The frontend then derives the displayed state from these. The visual card states you listed (CONNECTED / ACTIVE / AUTH FAILED / DISABLED / COMING SOON / PAPER MODE) are runtime states, not catalog properties — they come from joining catalog × userExchangeConnections × live health probe. Important to separate.
