Done. Stripe initialization no longer depends on Replit-only env vars.

What changed in artifacts/api-server/src/stripeClient.ts
getCredentials() was the hard-fail point — it threw "Missing Replit environment variables..." whenever REPLIT_CONNECTORS_HOSTNAME + REPL_IDENTITY/WEB_REPL_RENEWAL weren't both set, which is always the case on Render. Rewritten to a two-stage resolver:

1. credentialsFromEnv()           ← PRIMARY (Render & any non-Replit host)
     STRIPE_SECRET_KEY            (required)
     STRIPE_PUBLISHABLE_KEY       (optional; falls back to VITE_STRIPE_PUBLISHABLE_KEY)
2. credentialsFromReplitConnector()  ← LEGACY FALLBACK (Replit workspace only)
     returns null instead of throwing when REPLIT_* vars are absent

The connector path is no longer reachable on Render because env vars take precedence and the connector function returns null (not throws) when Replit-specific vars are missing. Final fallback error message is now actionable: "Set STRIPE_SECRET_KEY (and optionally STRIPE_PUBLISHABLE_KEY) in the deployment environment."

All three consumers benefit:

getUncachableStripeClient() — used by /billing/checkout, /billing/portal, /billing/subscription (routes/billing.ts:135, 397, 448)
getStripePublishableKey() — used by /billing/publishable-key (routes/billing.ts:153)
getStripeSync() — used by webhook handler (webhookHandlers.ts:18) and startup auto-registration (index.ts:35). Webhook secret now prefers STRIPE_WEBHOOK_SECRET env first; the stripe._managed_webhooks DB lookup only runs when env is unset (so Render won't try to query a table that doesn't exist there).
